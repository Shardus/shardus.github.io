# Shardus App Interface

The shardus interface is very easy to use, but requires an understanding of the various functions an app developer should be concerned with when seting up the network. You can always check out the very basic coin-app-template, or chat-app-template repo on gitlab for a full picture of some sample applications.

These functions should be specific to the type of app you wish to build, and the kind of transactions you expect the network to process. As an application developer, you won't need to worry about any of the internal complexity handled by shardus. You don't need to know about consensus, peer-to-peer communication, scaling, or sharding. However, if you do happen to know about these topics, you can further customize the configuration parameters to suit your needs. When shardus becomes open source, you would also be able to hot swap modules for these technical aspects to best suit your application.

# Interface

Shardus expects some important functions to be passed into the exposed setup method. After these functions have been implemented by the developer, call registerExceptionHandler and start

const shardus = require("shardus-enterprise-server-dist");

const dapp = shardus();

dapp.setup({
  validateTransaction(tx) {},
  validateTxnFields(tx) {},
  apply(tx, wrappedStates) {},
  getKeyFromTransaction(tx) {}
});

dapp.registerExceptionHandler();
dapp.start();

# validateTransaction

This function supplies the application with a validation checkpoint to ensure certain criteria are met before passing the transaction through to apply. The purpose of the function is to generate a response object with the result of the validation, and reasoning for that result. Essentially, this should make sure everything in the transaction is trusted and won't have any issues when it gets applied to your application state data.

// A failed response
const response = {
  result: "fail",
  reason: "Transaction is not valid."
};
// A successful response
const response = {
  result: "pass",
  reason: "Transaction is valid."
};

More details

# validateTxnFields

This function serves as an extra validation checkpoint for the fields passed into the transaction. It should check to make sure all the correct data types are passed in to the specific type of transaction that's about to be processed.

if (typeof tx.timestamp !== "number") {
  result = "fail";
  reason = '"timestamp" must be a number.';
  throw new Error(reason);
}

# apply

This function is responsible for handling the business logic that mutates your application state. If your application mutates state data anywhere else other than apply, the node will become out of sync and will be kicked out of the network. Every state change MUST be done within apply because all nodes see the transaction and make the same change within the apply function at the same time.

if (type === "transfer") {
  let sourceAccount = accountStates[tx.sourceAddress].data;
  let targetAccount = accountStates[tx.targetAddress].data;

  sourceAccount.balance = sourceAccount.balance - amount;
  targetAccount.balance = targetAccount.balance + amount;

  sourceAccount.timestamp = tx.timestamp;
  targetAccount.timestamp = tx.timestamp;
}

# getKeyFromTransaction

This function is responsible for parsing the public keys from the transaction for shardus to use. The source and target key types should be returned in a result object that shardus will use internally. For example, when sending a distribute type of transaction, the source keys should be an array containing the public key of the source account from which the coins are being sent, and the target keys should be the public keys of all the recipients of this transaction.

switch (tx.type) {
  case "transfer":
    result.sourceKeys = [tx.srcAcc];
    result.targetKeys = [tx.tgtAcc];
    break;
  case "distribute":
    result.sourceKeys = [tx.srcAcc];
    result.targetKeys = tx.tgtAccs; //tgtAccs could exist as an array on the transaction
    break;
}
